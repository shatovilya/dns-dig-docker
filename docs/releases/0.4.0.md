# DNS Debug v0.4.0 — PostgreSQL Historical Persistence

**Release date:** 2026-07-10  
**Semver:** minor (0.3.x → 0.4.0)  
**Tag:** `v0.4.0`

## Summary

v0.4.0 adds **local PostgreSQL** as the primary persistence layer for historical UI data with a **7-day retention policy**. Live DNS engine, Prometheus metrics, core API, and optional Web UI contracts remain backward compatible. When `DNS_DEBUG_DB_ENABLED=false`, file-based snapshots (`SNAPSHOT_*`) continue to work as in v0.3.0.

Key capabilities:

- PostgreSQL-backed snapshots and normalized aggregates (runs, resolver/domain/error/EDNS, MTR, chart buckets)
- Automatic retention cleanup (startup + periodic)
- Docker Compose `postgres` service with healthcheck and persistent volume
- Additive UI envelope fields for retention transparency
- File snapshot import on startup for migration from JSON files

## UX changes

### Historical mode

- Snapshot list and detail read from PostgreSQL when DB enabled (filtered to retention window)
- Retention banner: "Historical data retained for N days in local PostgreSQL"
- New warnings: `outside_retention_window`, `db_unavailable`
- `storage_backend: postgres|file` additive field when `snapshot_id` is set
- Compare mode unchanged (server-side deltas); snapshot picker respects 7-day window

### Unchanged

- Live mode pipeline (in-memory `stats_store`, Prometheus)
- Event buffer `from`/`to` historical path (in-process only; not persisted to PG)
- Read-only UI default (`DNS_DEBUG_UI_READONLY=true`)

## Configuration / migration

### New env vars (`DNS_DEBUG_DB_*`)

| Variable | Default (compose) | Description |
|----------|-------------------|-------------|
| `DNS_DEBUG_DB_ENABLED` | `true` | Enable PostgreSQL persistence |
| `DNS_DEBUG_DB_HOST` | `postgres` | Database host |
| `DNS_DEBUG_DB_PORT` | `5432` | Database port |
| `DNS_DEBUG_DB_NAME` | `dns_debug` | Database name |
| `DNS_DEBUG_DB_USER` / `PASSWORD` | `dns_debug` | Credentials (env only; not logged) |
| `DNS_DEBUG_DB_SSLMODE` | `disable` | SSL mode for local dev |
| `DNS_DEBUG_DB_RETENTION_DAYS` | `7` | Retention window |
| `DNS_DEBUG_DB_CLEANUP_ENABLED` | `true` | Enable automatic cleanup |
| `DNS_DEBUG_DB_CLEANUP_INTERVAL_SECONDS` | `3600` | Periodic cleanup interval |
| `DNS_DEBUG_DB_IMPORT_FILES_ON_STARTUP` | `true` | Import JSON from `SNAPSHOT_DIR` |

### Migration notes

1. `docker compose up -d --build` starts `postgres` + `dns-debugger`
2. Existing JSON snapshots in `./data/snapshots` are imported idempotently on first startup
3. Set `DNS_DEBUG_DB_ENABLED=false` to keep file-only mode without Postgres
4. Schema applied automatically via `app/db/migrations/001_initial.sql` at startup

### Database tables

`historical_snapshots`, `test_runs`, `run_aggregates`, `resolver_aggregates`, `domain_aggregates`, `error_aggregates`, `edns_aggregates`, `mtr_runs`, `chart_buckets`, `schema_migrations`

## Known trade-offs

- Raw query events are **not** persisted (volume); historical `from`/`to` without `snapshot_id` uses in-memory event buffer for the active process only
- Snapshot-based historical and compare modes are the primary 7-day UX path
